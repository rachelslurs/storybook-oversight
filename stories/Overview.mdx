This Storybook **dogfoods the addon**. `@storybook/addon-mcp` serves the
components manifest, and **Oversight** lints it, flagging the documentation an
AI agent can and can't read, per component, while you work.

Everything below is a real, live diagnosis: `@storybook/addon-mcp` generates the
manifest, Oversight reads it, and the panel and block react to what each
component actually documents.

## Two places to look

The same diagnostics appear in two independent surfaces.

### 1 · The panel

An **Oversight** tab in the addons drawer, on every component's **Story** view,
with a count badge of how many issues it found. Pick a component in the sidebar,

## The examples

Each component is deliberately built to trip **one rule**, so you can see the
panel and block react. Open each one and watch the Oversight tab (Story view) or
scroll to the block (Docs view):

- **[Button](?path=/docs/actions-button--docs)**: fully documented, and its
  description ends with a valid `?path=` redirect to Card → **clean**, and the
  link renders clickable in the panel.
- **[Card](?path=/docs/data-display-card--docs)**: an undocumented **required**
  prop → `required-prop-undocumented` (error) plus `prop-descriptions-missing`
  (warning). _(pictured above)_
- **[Badge](?path=/docs/data-display-badge--docs)**: no component description →
  `component-description-missing` (warning).
- **[Tile](?path=/docs/data-display-tile--docs)**: a `?path=` link pointing at a
  component that isn't in the manifest → `docs-link-dangling` (error).
- **[Banner](?path=/docs/feedback-banner--docs)**: a `@deprecated` tag →
  `deprecated-tag` (info).
- **[Panel](?path=/docs/layout-panel--docs)**: `@oversightIgnore` exempts its
  prop-coverage rule, but the list also names a rule that doesn't exist →
  `unknown-ignore-rule` (warning).

## Why these are rules, not a manifest view

`@storybook/addon-mcp` already lets you _view_ the raw manifest. Oversight earns
the panel by making calls a viewer can't:

- **`extractor-drift`** is a comparison. The manifest looks fine on its own; it's
  only wrong relative to the extractor you expected.
- **`docs-link-dangling`** takes every id in the manifest to know a `?path=`
  redirect points at nothing. One component's entry can't tell (the panel strikes
  the dead link through where it appears in the description).
- **`required-prop-undocumented` vs `prop-descriptions-missing`**: every blank
  prop reads the same in a raw view. Oversight calls the required one an `error`,
  because that's the gap an agent is most likely to guess through.

The `error` / `warning` / `info` severities are exactly that judgment: how likely
a gap is to mislead an agent. A viewer can show you the manifest; only a linter
has an opinion about it.

## The manifest-level rule

`extractor-drift` isn't tied to one component. It fires when the manifest was
built with a different docgen extractor than you expect (`react-docgen-typescript`
by default). The shallow `react-docgen` parser silently drops complex prop types,
so descriptions come back empty across the board. Because it's a property of the
whole manifest, the panel gives it its own **Manifest** section, above the
per-component findings:

<img
  src={driftShot}
  alt="The Oversight panel's Manifest section flagging extractor-drift, above the component's findings"

This demo ships on `react-docgen-typescript`, so drift can't fire live — which is
the point: the docs are intact. The still above is a build with the shallow
extractor; in your project the warning appears the moment the manifest drifts.

## Not shown here

`docgen-missing` and `story-extraction-error` fire only on extraction
**failures**, which can't be staged with well-formed components. They're covered
by the addon's unit tests instead.
